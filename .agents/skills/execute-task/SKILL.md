# Superpowers 任务执行技能 (Task Execution Skill)

## 触发条件 (Trigger)
当用户输入以下类似指令时，你**必须**强制调用此技能：
- "开始执行任务"
- "按 tasks 开始写代码"
- "应用变更 (Apply)"
- "执行下一个 task"

## 前置要求 (Prerequisites)
1. 当前工作区必须存在 `tasks.md` 文件，且包含未完成的任务 (`- [ ]`)。
2. 必须已完整读取对应变更目录下的 `proposal.md` 或 `design.md` 获取上下文。

## 核心执行指令 (Instructions)
当你被触发执行开发任务时，**绝对禁止**直接输出或写入大量业务代码。你必须严格按以下四个微观步骤循环执行：

### 第一步：加载上下文与头脑风暴 (Brainstorm)
1. 读取 `tasks.md` 找到第一个未完成的 `- [ ]` 任务。
2. 结合规范上下文，在终端简短输出（限 100 字以内）：
   - 实现该任务的核心思路。
   - 考虑到的边界情况 (Edge Cases) 和潜在的代码破坏风险。
3. **强制拦截**：输出完成后，必须主动向用户提问：“我的思路是否正确？是否可以开始编写测试和代码？”。**等待用户明确同意后才能进入下一步。**

### 第二步：TDD 落地编码 (Red-Green-Refactor)
获得用户同意后，必须遵循测试驱动开发流程：
1. **RED**：先创建或修改测试文件，编写针对该任务注定会失败的测试用例（或严格的类型校验）。
2. **GREEN**：编写刚好能通过测试的最小必要业务逻辑代码。
3. **REFACTOR**：清理冗余代码，确保严格符合 `.agents/rules/` 目录下的项目规范。

### 第三步：强制双重自我审查 (Double Review)
代码编写完成后，在修改任务状态前，必须静默进行内部核对：
1. **架构对齐**：当前代码实现是否完全符合 `design.md` 和 `specs/` 中的预设要求？
2. **代码健壮性**：异常是否全部被正确捕获？类型定义是否严谨？

### 第四步：状态更新 (Update Status)
1. 确认自我审查无误后，修改本地的 `tasks.md` 文件，将刚才完成的任务状态精准更新为 `- [x]`。
2. 告知用户当前任务已完成，并主动询问：“是否继续执行下一个任务？”
