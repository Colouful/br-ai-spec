# Superpowers 技能调度规范

> 核心原则：**在执行任何操作之前，先检查是否有适用的技能。**

## 子代理豁免

如果你是被派遣执行特定任务的子代理（subagent），跳过本技能，直接执行任务。

## 强制约束（NON-NEGOTIABLE）

如果你认为哪怕有 1% 的可能性某个技能适用于当前任务，你**必须**先读取并调用该技能。

这不是建议，是强制要求。不可协商，不可跳过，不可自我合理化地绕过。

## 指令优先级

1. **用户显式指令**（AGENTS.md、直接请求）— 最高优先级
2. **Superpowers 技能** — 覆盖默认行为
3. **系统默认提示** — 最低优先级

如果用户指令与技能冲突，以用户指令为准。

## 如何使用技能

在不同 AI IDE 中，技能的加载方式不同，但核心原则一致：

| 环境 | 技能加载方式 |
|------|-------------|
| **Cursor** | 通过 `.cursor/skills/` 目录下的 `SKILL.md` 描述自动匹配触发，或用户通过 `@` 显式引用 |
| **Claude Code** | 使用 `Skill` tool 调用，技能内容自动加载 |
| **Gemini CLI** | 通过 `activate_skill` tool 激活 |
| **其他环境** | 使用 Read 工具读取 `.agents/skills/<name>/SKILL.md` 文件并遵循其指令 |

## 技能检查流程

收到用户消息后，按以下流程执行：

```
1. 收到用户消息
   │
2. 判断：是否有任何技能可能适用？（哪怕 1% 的可能性）
   │
   ├─ 否 → 直接回复（包括澄清问题）
   │
   └─ 是 → 读取对应技能文件
            │
3. 宣告："使用 [技能名] 来 [目的]"
   │
4. 技能中是否包含检查清单？
   │
   ├─ 是 → 为每项创建 TODO，逐项执行
   │
   └─ 否 → 严格遵循技能指令执行
```

## 自我检查：危险信号

当你产生以下想法时，**立即停下** — 你正在合理化跳过技能：

| 你的想法 | 事实 |
|---------|------|
| "这只是一个简单问题" | 问题也是任务，检查技能 |
| "我需要先了解上下文" | 技能检查在了解上下文**之前** |
| "让我先看看代码" | 技能会告诉你**怎么看** |
| "这个任务不需要正式技能" | 如果技能存在，就用它 |
| "我记得这个技能的内容" | 技能会迭代更新，读取最新版本 |
| "这不算一个任务" | 有操作就是任务，检查技能 |
| "技能太重了" | 简单任务也会变复杂，先用技能 |
| "我先做这一件事" | 做任何事之前先检查技能 |
| "我知道该怎么做" | 知道概念 ≠ 使用技能，调用它 |

## 技能优先级

当多个技能可能适用时，按以下顺序：

1. **流程类技能优先**（头脑风暴、调试）— 决定「怎么做」
2. **实现类技能其次**（组件创建、API 封装）— 指导「做什么」

示例：
- "开发 X 功能" → 先 `create-proposal`，再 `create-component` / `create-api`
- "执行 tasks.md" → 先 `execute-task`（Superpowers Loop）
- "修复这个 bug" → 先调试定位，再读对应领域技能

## 技能类型

**刚性技能**（如 `execute-task`、调试流程）：严格遵循，不可简化步骤。

**柔性技能**（如设计模式、最佳实践）：适配当前上下文，灵活应用原则。

技能文件本身会标明属于哪种类型。

## 用户指令的解读

用户指令说明的是「做什么」，不是「怎么做」。

"新增 X 功能" 或 "修复 Y 问题" **不代表**可以跳过工作流。仍需检查并使用适用的技能来执行。
