# 项目实操最小落地方案

## 1. 这份方案解决什么问题

针对当前项目 `AI 规范驱动开发 CLI`，当前最重要的不是继续扩展概念，而是先落一套：

- 可演示
- 可扩展
- 不复杂
- 层级不深
- 能和现有 `.agents`、`openspec/` 协同

的最小文件与流程方案。

它主要服务两件事：

1. 支撑团队内部“项目实操”分享
2. 为后续专家协同开发提供一个稳定、浅层的上下文底座

## 2. 当前项目应该怎么理解

当前项目不是两条独立产品线，而是一个产品的两层表达：

```text
AI 规范驱动开发 CLI
  -> 实操层：专家协同开发
  -> 方法层：AI 工程方法论
```

所以当前改造优先级应该是：

- 先把“专家协同开发”的最小闭环做出来
- 同时用最小方法论解释它为什么成立

不建议现在同时铺开完整平台、完整专家矩阵、完整方法论资产体系。

## 3. 最小可扩展文件结构

建议在目标项目中采用一个浅层目录：

```text
target-project/
├── .agents/
├── context/
│   ├── PROJECT.md
│   ├── RULES.md
│   ├── DESIGN.md
│   ├── TASKS.md
│   ├── ITERATIONS.md
│   └── CHECKLIST.md
└── openspec/                 # 可选；启用 L3 时存在
```

这个结构有 3 个特点：

- 只有一层 `context/`
- 文件职责清晰
- 后续要扩展也容易

## 4. 为什么先用 6 个 md，不上更复杂体系

当前阶段不建议一开始就上 7 个、10 个甚至更多文档文件。

先用 6 个就够了：

- `PROJECT.md`
- `RULES.md`
- `DESIGN.md`
- `TASKS.md`
- `ITERATIONS.md`
- `CHECKLIST.md`

原因：

- 够用，能支撑一个完整 demo
- 不会让项目结构变重
- 团队更容易接受
- 后续再加 `DECISIONS.md`、`TELEMETRY.md` 也很自然

## 5. 这 6 个文件分别承担什么角色

### `PROJECT.md`

放稳定背景，只改少量内容：

- 项目定位
- 业务目标
- 核心术语
- 角色模型
- 架构概览
- 非目标

### `RULES.md`

放硬约束：

- 目录规范
- 命名规范
- API 约束
- 样式约束
- 安全底线
- 提交和验证要求

### `DESIGN.md`

放当前需求设计，是这次变更的核心说明。

建议包含：

- 问题定义
- 目标与非目标
- 页面/功能设计
- 接口与数据流
- 风险

### `TASKS.md`

放执行拆解，尽量短而明确：

- 任务列表
- 执行顺序
- 依赖
- 验收点

### `ITERATIONS.md`

放关键过程信息，不记流水账，只记有价值的：

- 试过什么
- 为什么失败
- 为什么改方案
- 当前遗留风险

### `CHECKLIST.md`

放验证清单：

- lint
- typecheck
- test
- build
- review
- 性能
- 安全

## 6. 它和 spec 的关系

这 6 个 md 和 `spec` 不是两套竞争体系，而是同一套体系里的不同切面。

建议这样理解：

- `PROJECT.md` / `RULES.md` 是稳定上下文
- `DESIGN.md` / `TASKS.md` 是当前变更上下文
- `ITERATIONS.md` / `CHECKLIST.md` 是反馈和验证上下文

而 `spec` 主要相当于这套结构中的：

- `DESIGN.md`
- `TASKS.md`
- 部分关键决策说明

所以更准确的说法是：

> `spec` 是“当前变更”的核心文档，而不是整个项目上下文系统的全部。

## 7. 当前项目最适合演示的黄金路径

建议你分享时只演示一条路径，不要把所有能力一次讲完。

推荐黄金路径：

```text
输入 PRD / 设计
  -> 填写或生成 DESIGN.md
  -> 拆成 TASKS.md
  -> 前端实现专家开发
  -> 规范守护者检查
  -> CHECKLIST.md 验证
  -> ITERATIONS.md 记录结果
```

如果项目开启了 OpenSpec，则可以再补一句：

```text
openspec/changes/.../proposal.md / tasks.md
  -> 同步映射到 DESIGN.md / TASKS.md
```

这样就能把 “spec” 和 “项目上下文 md” 讲成一套，而不是两套。

## 8. 当前阶段建议只做 3 个专家

为了保证分享聚焦，建议先只演示 3 个专家：

1. `需求解析专家`
   - 输入 PRD / 设计
   - 输出 `DESIGN.md`

2. `前端实现专家`
   - 输入 `DESIGN.md` + `TASKS.md`
   - 输出代码实现

3. `规范守护者`
   - 输入实现结果
   - 输出规则检查和修正建议

这个组合最容易解释，也最符合当前项目定位。

## 9. 当前阶段不要先做什么

先不要优先做：

- 完整专家矩阵
- 多层复杂目录
- 完整可观测平台
- 完整角色权限系统
- 过多流程分支

这些方向都对，但不适合当前作为第一轮项目实操分享的重点。

## 10. 对当前项目的具体改造建议

围绕当前项目，建议先做下面这些最小改造：

1. 在仓库中提供 `templates/context-pack/`
   - 让团队直接看到推荐文件结构

2. 在文档里明确 `spec` 和 `context` 的关系
   - 避免团队误以为是两套体系

3. 分享只围绕一个 demo 场景
   - `PRD/设计 -> 开发 -> 校验 -> 记录`

4. 先用 3 个专家，别一开始铺太满

5. 先把 `context/` 做成浅层目录
   - 不要一开始做很深的嵌套

## 11. 最后的建议

如果以“项目实操分享”为目标，当前最优策略不是继续扩概念，而是：

> 用一套最小上下文包 + 一个黄金路径 demo + 三个核心专家，把 `AI 规范驱动开发 CLI` 讲清楚、跑通、讲透。

这比一开始就做成完整平台更稳，也更容易让团队理解和接受。
